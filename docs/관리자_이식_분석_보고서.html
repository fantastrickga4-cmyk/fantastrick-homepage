기존 사이트 예약 시스템의 정체

기존 관리자는 워드프레스 + Booked 예약 플러그인이었습니다. "예약확인" 메뉴가 곧 예약 운영의 심장입니다.

메뉴하는 일
예약확인월별 캘린더 뷰 + 테마별 필터(사자의 서·태초의 신부·락다운시티·시간의 영속성). 상단에 “SMS 남은 건수 : 2,759건” → 실제 문자 발송 중
입금확인입금 안 된 예약만 모아 승인 대기 큐로 관리
테마 설정테마=캘린더 4개. 누적 예약수: 태초의 신부 13,867 · 사자의 서 7,062 · 락다운시티 1,160 · 시간의 영속성 229
환경 설정요일별 타임슬롯 · 예약버퍼 · 예약 가능기간 · 취소마감 · 슬롯당 예약수 · 테마별×이벤트별 SMS 템플릿 · Custom Fields

그 외 Rooms(테마 상세: 난이도 Easy/Normal/Hard·인원·소요시간·평점·갤러리), 팝업 공지(“DLC 연기안내”·“공지사항”), 연락처 폼(CF7)이 있었습니다.

기능별 갭 & 적용 방안

1. 입금대기 전용 큐 P1부분있음

돈이 아직 안 들어온 예약만 따로 모아 “승인할까?”를 처리하는 대기줄.

우리 현재 · 상태필터 pending으로 거를 수 있고(‘확정 대기(미입금)’ 통계 + 30초 폴링 🔔새 예약 알림) 30분 뒤 자동취소(lib/expire.ts)까지 있음. 단 “미입금만” 콕 집는 전용 필터는 없음.

적용 · 새 탭 대신 reservations/route.ts GET에 deposit=unpaid 추가(.eq("deposit_paid",false)) + 예약탭에 “입금대기 N건” 버튼. 공수 小.

2. 요일별·매장별 타임슬롯 P1없음

기존은 “월요일은 이 시간들, 토요일은 저 시간들”처럼 요일·매장마다 시간표를 다르게 짤 수 있음.

우리 현재 · TIME_SLOTS 1세트가 모든 매장·모든 요일 공통. 그런데 2호점은 실제로 평일 부분운영이라 안 여는 날에도 예약칸이 뜸.

적용 · app_settings.time_slots를 {매장:{default,요일override}} 구조 JSON으로 확장 + slotsFor(store,date) 헬퍼 신설(공개 예약화면·검증·관리자 3곳 동시 반영). 우선 “매장별 1세트”부터, 요일별은 뒤로. 공수 中.

3~5. 예약버퍼 · 예약가능기간 · 취소마감 P2

기능우리 현재적용 방안
예약버퍼(임박 차단)부분있음 — 오픈규칙(이용 7일전 밤9시)으로 미래는 막히나 “당일 N시간 전” 하한은 없음reservationDateState에 minLeadMinutes 추가, 지난·임박 슬롯 제외
예약가능기간부분있음 — “일주일치만” 여는 방식이 방탈출에 오히려 적합필요 시 reserve_until 키 하나로 충분
취소마감없음 — 대신 환불율 규정(refund_rate)으로 처리취소 API에서 남은시간→차단 or 환불율 자동계산(규정표를 app_settings에)

참고 #6 슬롯당 예약수는 schema.sql의 uq_res_slot로 한 슬롯 1팀 전세 = 방탈출에 정확. 변경 불필요.

7 · 8 · 9. 문자(SMS) 3종 갭 P1P2

기존은 문자를 훨씬 세밀하게: 잔여건수를 화면에 띄우고, 테마별·상황별로 다른 문구를 보냄.

잔여건수(P2·小) · 우리 sms/route.ts엔 잔여 조회 없음 → 알리고 /remain/ 호출해 SmsTab 상단 배지.

이벤트 확장(P1·小) · 우리는 confirm/cancel/reminder 3종뿐, 기존은 예약/취소/입금/관리자취소 4종. “입금 확인됐습니다” 문자가 없음 → type에 payment·admin_cancel 추가하고 입금 토글 시 자동발송.

테마별 문구(P2·中) · sms_templates에 theme_id 컬럼 추가, getTemplate(type,themeId) 없으면 전역 폴백. 4테마라 실무상 이벤트확장이 먼저.

11. 테마 상세 메타 P2부분있음

테마 소개에 뜨는 난이도·인원·소요시간·별점·사진 여러 장.

기존 필드우리 대응상태
난이도 Easy/Normal/Harddifficulty:number있음(라벨매핑만)
소요시간minutes:number있음
인원(정원)—없음
평점리뷰 평균으로 산출 가능없음
갤러리(여러 장)poster 1장만없음

적용 · Theme 타입에 players?·gallery?[] 추가(코드 상수), 평점은 reviews 평균. 관리자보다 소개페이지 품질 이슈.

12. 팝업 공지사항 P2없음

홈 들어오면 뜨는 공지 팝업(휴무·이벤트 안내). 기존엔 “DLC 연기안내”·“공지사항”.

적용 · app_settings에 notice{enabled,title,body,until} 추가 → 설정탭에 입력 UI → 공개 api/config로 노출 → 홈에 <NoticeModal>(오늘 하루 안 보기 localStorage). 사장님이 코드 수정 없이 공지 가능. 공수 中.

10 · 13 · 14. 나머지 P3

누적 예약수 · 우리 buildStats.byTheme로 상대비교는 되나 새 DB라 0부터. 마케팅 문구 필요시 theme_base_count 시작값.

비회원 예약 · 이미 있음 — 전화+이름+PIN 4자리가 Booked Guest보다 단순·안전. 추가 불필요.

Custom Fields · 예약폼에 자유 request 필드 하나면 충분(완전 동적 필드빌더는 과함).

권장 실행 순서

🟥 스프린트 1 — P1 · 하루 운영 필수

  1. 입금대기 필터(#1): deposit=unpaid + 예약탭 버튼 [小]
  2. 문자 이벤트 확장(#9): payment·admin_cancel + 입금 토글 자동안내 [小]
  3. 매장별 타임슬롯(#2 1단계): 2호점 평일 오예약 차단 [中]

🟧 스프린트 2 — P2 · 있으면 좋음

  1. 팝업 공지사항(#12) — 사장님 셀프 공지 [中]
  2. SMS 잔여건수(#7) + 예약버퍼 하한(#3) + 취소마감/환불율 자동계산(#5) [小~中]
  3. 테마별 문자 문구(#8) + 요일별 슬롯(#2 2단계) [中]

🟩 스프린트 3 — P3 · 여유될 때

  1. 테마 메타 보강(정원·갤러리·평점) · 예약폼 요청사항 필드 · 테마 누적 시작값

굳이 이식 안 해도 되는 것

워드프레스 특유의 인프라라, 우리 스택엔 필요 없거나 이미 다른 방식으로 해결된 것들.

  • Elementor / Slider Revolution — WP용 페이지빌더·슬라이더. 우리는 React 컴포넌트로 직접 제작.
  • WP File Manager — 서버 파일 브라우저. Next.js엔 개념 없음(보안상 안 만드는 게 나음).
  • Instagram Feed / Simple Ajax Chat — 임베드 한 줄이면 되고 관리자 기능 아님.
  • WPS Hide Login / Redirection / WP Mail SMTP — 우리 admin/login(쿠키 세션)·Vercel 라우팅으로 이미 해결.
  • ACF PRO / Custom Post Type(Rooms) — WP 데이터 모델링 도구. 우리는 data.ts 상수 + Supabase 테이블이 그 역할.
  • Front-End Agents / Calendar Feeds — Booked 부가기능. 1인 매장엔 과함.
결론 — 핵심 예약운영 3가지(입금대기 필터 · 입금/관리자취소 문자 · 매장별 슬롯)만 P1으로 먼저 채우면 Booked 대비 실사용 갭은 대부분 메워집니다. 나머지(요일별 슬롯·테마별 문자·팝업 공지)는 “있으면 좋음” 수준이라 스프린트 2~3으로 미뤄도 무방합니다. 워드프레스의 페이지 편집·플러그인 인프라는 옮길 필요 없습니다.
